# Script Review

用于审查现成逐字稿或生成后的逐字稿。

## Review 立场

优先发现会影响考场表现的问题，而不是泛泛表扬。问题按严重程度排序，并给出可执行修改方向。

## 严重程度

- High：会明显破坏试讲标准、导致超时/过短、结构缺失、知识逻辑错误、互动失真或语言严重不自然。
- Medium：会降低课堂质量，如评价空泛、过渡生硬、板书不清、重点分散、思政突兀。
- Low：局部表达、格式或细节优化，不影响整体成立。

## 检查项

### 结构与时间

- 是否完整包含五个环节。
- 是否控制在 950 到 1100 字。
- 重点知识数量是否与材料强度、逻辑连贯性、试讲容量和逐字稿字数匹配。
- 是否存在导入过长、新授不足、练习过重、结尾拖尾等节奏问题。

### 内容与逻辑

- 是否有清晰主线。
- **是否有明确点明重难点的话术**（如：“这是本节课的核心/重点/难点”），而不是只在结构上划分。
- 是否有知识建构，而不只是罗列概念。
- 是否轻重得当，有没有面面俱到或避重就轻。
- 是否存在不符合网络工程常识或教材逻辑的问题。

### 互动与评价

- 互动是否服务于知识建构。
- 学生回答后是否有请坐、复述提炼、自然推进。
- 评价是否具体，是否避免“很好”“非常棒”等模板话。
- 互动形式是否与问题难度匹配。

### 语言与排版

- 是否口语化、自然、有课堂感。
- 是否有动作标签或舞台提示。
- 是否存在第二人称复述学生回答的问题。
- 板书是否只写关键词，是否随讲随写。

### 练习、思政与作业

- 巩固练习是否简短快速（约 45 秒），题目是否避免了复杂情境分析。
- 小结是否先学生总结再教师提升。
- 思政是否自然融入且不喧宾夺主。
- 作业是否采用必做和选做两档。

## 输出格式

Review 输出按以下结构：

1. 主要问题：按 High、Medium、Low 排序。
2. 修改建议：给出可直接执行的修订方向。
3. 保留亮点：只列真正值得保留的设计。
4. 是否建议重写：说明是局部修订、结构性重写，还是只需微调。
